 82  # Use an integer.

 86  # Array sizes: grid_par['nx','ny','nz'] 
 87  # 
 88  # The code works for 1D, 2D, or 3D simulations.  For 2D or 1D set one 
 89  # or two of the array dimensions to 1. 
 90  # 
 91  # Use only even values (or 1) for nx, ny, and nz.  I'm not convinced 
 92  # the code is working correctly for odd values. 
 93  # 
 94  # The FFTs will be most efficient if the array sizes can be factored 
 95  # into combinations of 2, 3, 4, and 5.  Nevertheless, the FFTs may 
 96  # still be fast if a remaining factor bigger than 5 is not too large. 
 97  # 
 98  # Use integers.

102  # Cell sizes: grid_par['dx','dy','dz'] 
103  # 
104  # You may use integers or floats, conversion to float is done for 
105  # you when necessary. 
106  # 
107   
108  # 
109  # Covariance grid cosine taper switch: covar_par['taper_switch'] 
110  # 
111  # A cosine taper will be applied to the edges of the grid if 
112  # taper_switch is true, otherwise not. 
113  # 
114  # The default is to use a cosine taper, but some covariance models work 
115  # better without it.  An example of a covariance model that works 
116  # better without a taper is the stable model with small alpha and with 
117  # grid-aligned anisotropy.  When alpha is small the covariance can 
118  # decay very slowly so that it still has significant values, but small 
119  # derivatives on the edges of the grid. Because the derivatives are 
120  # small a taper is not needed.  However, if the cosine taper is applied 
121  # anyway, the covariance will end up with an unintended dominant cosine 
122  # shape on the edges, producing unexpected results. 
123  # 
124  # Anisotropic covariance models not aligned with the grid are more 
125  # difficult.  These models have a discontinuity on the grid edges 
126  # because of the spatial periodicity of the FFT which is aligned with 
127  # the grid, not the anisotropy.  This discontinuity will also produce 
128  # grid-aligned linear artifacts.  I don't know a good way to make 
129  # simulations with small alpha and non-grid-aligned anisotropy yet. 
130  # There are some situations where it seems you must put up with one 
131  # kind of artifact or another depending on whether tapering is turned 
132  # on or not.  It might help to apply averaging at the discontinuities, 
133  # but I haven't tried that yet. (see Briggs and Henson, The DFT: An 
134  # Owner's Manual for the Discrete Fourier Transform, SIAM, look in the 
135  # index for AVED, "average values at endpoints and discontinuities.") 
136  # 
137  # Some other time I might think of a better way of tapering that is 
138  # more robust and requires less user knowledge... 
139  #

141  # 
142  # Covariance range orientation vectors: covar_par['oriu','oriv','oriw'] 
143  # 
144  # The orientation vectors define a possibly rotated and skewed 
145  # coordinate system orienting the covariance model.  The correlation 
146  # ranges will be specified along these vectors. 
147  # 
148  # The orientation vectors do not need to be unit length, they will be 
149  # scaled.  The orientation vectors do not need to be orthogonal, 
150  # allowing for skewed covariance models. 
151  # 
152  # You may use integers or floats, conversion to float is done for you 
153  # when necessary. 
154  # 
155  # Anisotropic, orthogonal, and rotated example 
156  # oriu = [ 5,1,0] 
157  # oriv = [-1,5,0] 
158  # oriw = [ 0,0,1] 
159  # 
160  # Isotropic example 
161  # oriu = [1,0,0] 
162  # oriv = [0,1,0] 
163  # oriw = [0,0,1] 
164  # 
165   
166  # 
167  # Covariance range parameters: covar_par['ru','rv','rw'] 
168  # 
169  # Each covariance range parameter controls the correlation range in the 
170  # direction of the corresponding orientation vector. 
171  #  
172  # The range parameters should be small enough so that the derivative of 
173  # the covariance decays to "small" values at the edges of the grid. 
174  # This reflects the intuitive idea that it does not make sense to model 
175  # correlations that go beyond the edge of the grid.  The edges of the 
176  # covariance model can be tapered to prevent some of the artifacts that 
177  # would otherwise be present, but range parameters that are too large 
178  # will produce a covariance dominated by the taper rather than the 
179  # desired model. 
180  # 
181  # You may specify integers or floats, conversion to float is done for 
182  # you when necessary. 
183  # 
184   
185  #  
186  # "Stable" covariance model shape parameter: covar_par['alpha'] 
187  # 
188  # The stable covariance model for unit variance is: 
189  # 
190  # cov(h) = exp(-(h/r)^alpha) 
191  # 
192  # where h is a separation distance, or "lag," in a certain direction, r 
193  # is the corresponding range parameter, and alpha is the shape 
194  # parameter. 
195  # 
196  # The corresponding stable semi-variogram model is: 
197  # 
198  # gamma(h) = 1-exp(-(h/r)^alpha) 
199  # 
200  # The stable covariance model is asymptotically fractal (power law) 
201  # for small enough h: 
202  # 
203  # cov(h->small) = (h/r)^alpha 
204  # 
205  # For larger values of h, cov(h) approaches zero and gamma(h) 
206  # approaches a "sill" of one. 
207  # 
208  # The shape parameter alpha must be in the range 0 <= alpha <= 2.  The 
209  # special cases of 0, 1, and 2 correspond to the familiar "Nugget", 
210  # "Exponential", and "Gaussian" models of geostatistics.  These models 
211  # produce different amounts of "roughness" in the correlated field. The 
212  # Nugget model produces rough spatially uncorrelated random fields that 
213  # are discontinuous everywhere (Gaussian white noise).  The Exponential 
214  # model produces spatially correlated and continuous random fields with 
215  # intermediate roughness and first derivatives that are discontinuous 
216  # everywhere. The Gaussian model produces smooth spatially correlated 
217  # random fields with continuous derivatives.  Fractional values of 
218  # alpha are allowed, producing random fields anywhere in this spectrum 
219  # of roughness. 
220  #  
221  # You may specify an integer or a float, conversion to float is done 
222  # for you when necessary. 
223  #

230  # Compute the grid origin. 
231  # 
232  # To construct the random field we will need to sample the covariance 
233  # model on a grid and compute its FFT (among other things).  The FFT 
234  # expects spatially periodic data with "wrap-around" storage.  That is, 
235  # the origin of the coordinate system, (x,y,z)=(0,0,0), is expected to 
236  # coincide with the corners of the grid, and the parts of the 
237  # covariance model corresponding to negative values of x, y, or z are 
238  # expected to be stored in the "second half" of the x, y, or z axis. 
239  # 
240  # However, this storage order is non-intuitive and inconvenient for 
241  # sampling the covariance and taper functions. 
242  # 
243  # So instead we will set the values of (x,y,z) at the grid origin so 
244  # that (0,0,0) is in the center of the array.  This results in more 
245  # intuitive covariance and taper models centered in the middle of the 
246  # grid and more convenient sampling.  Later, we will rotate the tapered 
247  # covariance grid to shift the origin back into the corners of the 
248  # grid.  
249  # 
250   
251      grid_par['ox'] = -float(grid_par['nx']/2)*grid_par['dx'] 
252      grid_par['oy'] = -float(grid_par['ny']/2)*grid_par['dy'] 
253      grid_par['oz'] = -float(grid_par['nz']/2)*grid_par['dz']

262                 n3=%(nz)d d3=%(dz)g o3=%(oz)g 
263                 label1="x" label2="y" label3="z" 
264                 ''' % (grid_par)

394  # 
395  # Compute the FFT of the covariance and random noise. 
396  # 
397  # The Fourier transform of the covariance is the power spectrum.  
398  # Because the covariance should be real and symmetric (an even 
399  # function), the power spectrum should have no imaginary component, 
400  # except for roundoff.  If there is a significant imaginary component 
401  # then the covariance does not have the proper symmetry, or has not 
402  # been rotated correctly (anything else?). 
403  # 
404  # Also, the power spectrum should be everywhere positive, except for 
405  # roundoff.  If not, the covariance is not a valid positive-definite 
406  # model.  A non-positive-definite covariance implies a complex valued 
407  # random field. 
408  # 
409   
410      pspec       = real_par['name']+'pspec' 
411      pspec_real  = real_par['name']+'pspec_real' 
412      noise_fft   = real_par['name']+'noise_fft'

414      Flow (pspec,rotate, 
415            ''' 
416            rtoc |  
417            fft3 pad=1 opt=n axis=1 |  
418            fft3 pad=1 opt=n axis=2 |  
419            fft3 pad=1 opt=n axis=3 | 
420            put label1="wx" label2="wy" label3="wz" 
421            ''') 
422       
423      Flow (pspec_real,pspec,'real')

425      Flow (noise_fft,noise, 
426            ''' 
427            rtoc |  
428            fft3 pad=1 opt=n axis=1 |  
429            fft3 pad=1 opt=n axis=2 |  
430            fft3 pad=1 opt=n axis=3 | 
431            put label1="wx" label2="wy" label3="wz" 
432            ''')

434  #  
435  # Compute the amplitude spectrum. 
436  #  
437  # The amplitude spectrum we will use here is simply the square root of 
438  # the power spectrum.  This selects the unique real-valued amplitude 
439  # spectrum from the family of possible complex-valued and 
440  # Hermite-symmetric amplitude spectra. 
441  # 
442  # If the power spectrum has any negative values, corresponding to an 
443  # invalid non-positive-definite covariance model, then the amplitude 
444  # spectrum will end up complex valued, but not Hermite symmetric, 
445  # implying a complex-valued random field. 
446  #  
447  # The stable covariance model is positive definite for 0 <= alpha < 2. 
448  # The Gaussian model, alpha=2 is on the boundary of positive 
449  # definiteness.  It is not positive definite, but it is non-negative 
450  # definite.  Values larger than 2 produce negative definite models. 
451  # The use of a Gaussian convariance model with a discrete Fourier 
452  # transform will produce some small nagative values due to round off, 
453  # but these can be clipped without introducing significant artifacts. 
454  # 
455  # There are many possible ways we could deal with imaginary and/or 
456  # negative values in the power spectrum.  Here we just discard the 
457  # imaginary part and clip any real values below zero.  Thus, there 
458  # is nothing to stop the user from specifing a non-positive-definite 
459  # covariance, but if he does it will truncated in Fourier space 
460  # into something that is positive definite. 
461  # 
462  # Additional experimentation might turn up better methods for dealing 
463  # with complex or negative power spectrum values. 
464  # 
465   
466      aspec      = real_par['name']+'aspec' 
467      aspec_real = real_par['name']+'aspec_real' 
468   
469      Flow (aspec_real,pspec_real,'clip2 lower=0 | add sqrt=1') 
470      Flow (aspec,     aspec_real,'rtoc') 
471   
472  # 
473  # Combine the amplitude spectrum and noise. 
474  # 
475  # Multiplication in Fourier space is the same thing as convolution in 
476  # real space.  Thus, this step is the same thing as performing a 
477  # weighted moving average of the uncorrelated Gaussian noise, producing 
478  # a correlated Gaussian noise. 
479  # 
480  # If we wanted to see the shape of the moving average weight function 
481  # implied by this Fourier space convolution, we could calculate the 
482  # inverse Fourier transform of the amplitude spectrum.  The weight 
483  # function is sometimes called the "covariance square root" or 
484  # "convolution square root" of the covariance function.  That is, the 
485  # weight function is a function which when convolved with a reversed 
486  # copy of itself produces the desired covariance function. 
487  # 
488  # Because we selected the real-valued amplitude spectrum, rather than a 
489  # more general complex-valued and Hermite-symmetric amplitude spectrum, 
490  # the weight function is symmetric.  Otherwise the weight function 
491  # could be asymmetric, which might be interesting to tinker with some 
492  # day... 
493  # 
494   
495      sim_i_fft = real_par['name']+'sim_%02d_fft' 
496   
497      for i in range(real_par['nr']):  
498          Flow (sim_i_fft % (i+1),[noise_fft,aspec], 
499                ''' 
500                window squeeze=n f4=%d n4=1 |  
501                add mode=m ${SOURCES[1]} 
502                ''' % (i) ) 
503   
504  # 
505  # Invert the simulation FFT. 
506  # 
507  # The resulting stochastic simulation will have both real and imaginary 
508  # parts.  However, the imaginary part is produced by roundoff and should 
509  # be small in magnitude. 
510  # 
511   
512      sim_i      = real_par['name']+'sim_%02d' 
513      sim_i_real = real_par['name']+'sim_%02d_real' 
514   
515      for i in range(real_par['nr']):  
516          Flow (sim_i % (i+1),sim_i_fft % (i+1), 
517                ''' 
518                fft3 inv=y pad=1 opt=n axis=3 | 
519                fft3 inv=y pad=1 opt=n axis=2 | 
520                fft3 inv=y pad=1 opt=n axis=1 | 
521                put label1="x" label2="y" label3="z" 
522                ''') 
523       
524          Flow (sim_i_real % (i+1),sim_i % (i+1), 
525                'real | put o1=0 o2=0 o3=0')
